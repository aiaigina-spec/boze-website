## 二、Firebase 跟 GitHub 到底差在哪（白話比喻）

- **GitHub = 食譜本 + 修改紀錄（後台倉庫）**
  - 保存所有原始檔案，記得「哪天改了什麼、改壞了能還原」。
  - 它本身**不是拿來給訪客看網站的**。

- **Firebase Hosting = 開門營業的店面（前台）**
  - 真正讓訪客打開網址、看到網站的地方。

- **GitHub Pages = 讓倉庫「兼差」也開一間店面**
  - 所以我才會有兩個能看的網址。同一份原始碼，開了兩間一樣的店。

一句話：**GitHub 存檔、Firebase 營業。** 兩者是不同工種，不是二選一。

## 三、日常更新操作小抄（改完網站照抄這段）

改完 `boze-website` 資料夾裡的檔案後，跑以下指令，**一次同步更新兩個網站**：

```bash
# 1. 進到網站資料夾
cd ~/Documents/Claude/Projects/博策網站製作/boze-website

# 2. 更新 Firebase（主網站）
firebase deploy --only hosting

# 3. 更新 GitHub Pages（備援網站）＋ 存檔備份
git add -A
git commit -m "更新內容"   # 引號裡改成這次改了什麼
git push
```

- `git push` 之後，GitHub Pages 會**自動**重新建置，約等 1～2 分鐘才會看到更新，不用手動操作。
- 專案已經綁好，不需要再跑 `firebase use`。

## 四、絕對不能忘的三條紅線

1. **GitHub repo 目前是「公開」的** → 原始碼任何人都看得到。
   這個資料夾**永遠不要放** API 金鑰、後台密碼、客戶名單等機密資料。
2. **對外只報 Firebase 網址**（`boze-web-2026.web.app`）→ 短、好記，之後要綁自訂網域（如 `bozepartners.com`）也是從 Firebase 綁最順。
3. **改網站一定要「兩邊都更新」** → 只跑 `firebase deploy` 會讓 GitHub Pages 那邊變舊；
   只跑 `git push` 會讓 Firebase 那邊變舊。**兩段指令都要跑。**
